# Claims Map — A Brief Examination of Rubber Ducks in Software Engineering

The article makes five claims. Each is referenced in the body via `[claim:N]`
markers and grounded in cited prior work.

- [claim:1] Rubber duck debugging is a documented programming practice, with
  a clear lineage from informal lore to mainstream developer literature.
  *Grounding:* [@hunt1999pragmatic] popularized the term; the practice predates
  the publication.

- [claim:2] The mechanism is articulation, not the duck. The act of putting
  implicit understanding into explicit language is what surfaces the bug.
  *Grounding:* [@naur1985programming] frames programming as theory-building;
  articulation makes the theory inspectable.

- [claim:3] Cognitive offloading via designed objects has a deeper history than
  the duck — handles, signs, and other affordances do similar work.
  *Grounding:* [@norman1988design] on the role of objects in distributing
  cognitive load.

- [claim:4] Pair programming and code review are kin to duck debugging — the
  audience changes, but the requirement to articulate is the load-bearing
  factor. *Grounding:* [@brooks1995mythical] on the social dynamics of
  programming work.

- [claim:5] Literate programming is duck debugging written down — the
  documentation IS the articulation, made permanent. *Grounding:*
  [@knuth1984literate] on weaving prose and code as a single artifact.
