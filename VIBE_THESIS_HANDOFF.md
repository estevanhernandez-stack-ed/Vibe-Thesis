# Vibe Thesis — handoff sketch for Vibe Cartographer

Working brief for packaging the toolchain demonstrated in this repo
(`agentic-architect-vibe`) as a distributable **Claude Code plugin**
called **Vibe Thesis**.

> **Shape**: Vibe Thesis is a Claude Code skill bundle (not a CLI tool).
> The user invokes it inside Claude Code, converses with Claude about
> their subject, and Claude orchestrates: scaffolds the workspace,
> guides through proposal → research → drafting → rendering, and
> dispatches the supporting sub-skills (bootstrap, agent swarms,
> lay-translator, research-integrate, merge-authors) at the right moments.
> The render pipeline is still bash-callable, but the user never has to
> remember `npm run render:pdf` — they say "render the PDF" and Claude does.
>
> **End state for the user**: pick a subject, iterate with Claude in the
> repo, end up with a publishable thesis-shaped artifact (PDF + HTML +
> lay-version + manifest).

This document is the result of a decomposition pass over the current repo:
file inventory, commit-trace of where each piece came from, coupling map,
and a proposed MVP cut. It is not a plan — it is the raw material a planner
(Vibe Cartographer) needs to draft one.

## 4. Plugin architecture — Claude Code skill bundle

Claude Code recognizes several layered constructs:

- **Skill** — a single `SKILL.md` (YAML frontmatter + body) that Claude
  auto-loads when relevant. Already used in this repo for `bootstrap`,
  `lay-translator`, `merge-authors`, `research-integrate`.
- **Slash command** — a file under `.claude/commands/` invoked by the
  user typing `/<name>`. Good for explicit verbs.
- **Subagent** — a specialized agent with its own tool whitelist.
- **Plugin** — a bundle that can ship multiple skills + commands +
  subagents + settings + hooks together, distributed via a marketplace
  or git URL.

Vibe Thesis is naturally a **plugin** because it ships:

- 1 orchestrator skill (the entrypoint — the "vibe coach")
- 4 supporting skills (bootstrap, merge-authors, lay-translator, research-integrate)
- 1+ slash commands (e.g. `/vibe-render`, `/vibe-translate`, `/vibe-status`)
- A `templates/` payload — the workspace files Claude copies into the
  user's directory on first run (numbered dirs, render scripts, schemas,
  Dockerfile, `tokens.yaml`, `.gitattributes`, etc.)
- Default settings + hooks (e.g. allowlisted `npm run render:*` so the
  user isn't permission-prompted constantly)

> **Cartographer should verify the current Claude Code plugin format
> against the official docs before drafting the manifest** — the plugin
> system has been moving quickly and field shapes change. This sketch
> describes the intent; the exact `plugin.json` (or equivalent) layout
> is Cartographer's call to nail down against current docs.

### How the orchestrator skill works (sketch)

The `vibe-thesis` skill's `SKILL.md` is the brains. Pseudocode of its
decision tree:

```
when invoked or detected:
  if cwd has no plugin scaffold:
      → run scaffold mode:
          1. copy templates/ into cwd
          2. invoke bootstrap skill (interview user, fill placeholders)
          3. set THESIS_MODE based on user's answer
          4. offer to dev-container or native-install
          5. confirm npm run render:pdf works end-to-end
  else (project exists):
      → run iterate mode:
          • read 01_PLANNING/{proposal,outline,claims-map}.md
          • figure out which phase user is in (planning → research →
            drafting → review → translation)
          • coach in that phase: ask the next-most-useful question,
            propose the next-most-useful action
          • dispatch sub-skills at the right moments:
              - "I read this paper" → research-integrate
              - "let's strengthen claim X" → lit-review-swarm
              - "make this accessible" → lay-translator
              - "we have multiple authors" → merge-authors
          • offer renders + sync checks at natural breakpoints
```

The user is never told to "run command X" — they say what they want
in natural language and Claude bridges to the right skill, command, or
bash invocation.

### Why this shape over a CLI

The current repo's commit history *demonstrates* this shape works: the
user (Estevan) drove the entire writing process by talking to Claude,
and Claude invoked `npm run render:pdf` etc. as needed. The dev
container, render scripts, validation — they all exist as substrate
for Claude to use, not as commands the user types. Vibe Thesis just
generalizes that experience and ships it as a one-install-then-vibe
package.

A CLI would be valuable as a *secondary* surface (CI, headless renders,
non-Claude users), but it is not the primary product.

## 5. Proposed MVP cut

**Goal**: a user installs the Vibe Thesis plugin, opens a fresh
directory in Claude Code, types something like "I want to write a
thesis on X" — and 30 minutes later has a styled PDF skeleton with
proposal, outline, claim map, the dev container running, and a real
draft underway.

**In MVP**:

- Orchestrator skill (`vibe-thesis`) with scaffold mode + iterate mode
- Templates payload — the full numbered scaffold + render scripts +
  schemas + Dockerfile + design system, lifted directly from this
  repo's `template/main`
- Sub-skills: `bootstrap`, `merge-authors`. Both already in this repo,
  battle-tested.
- Slash commands: `/vibe-render`, `/vibe-status`
- Dev container with the three hard-won fixes already shipped (font
  install, `.gitattributes` LF normalization, persistent `~/.claude`
  volume mount)
- Plugin manifest + install instructions
- One worked example bundled: e.g. a 3-page demo article in `examples/`
  that round-trips through the whole pipeline so users can see what
  "done" looks like

**Out of MVP** (move to v0.2+):

- `lay-translator` and `research-integrate` sub-skills. They're real,
  but each is a deep workflow on its own — ship them as the next layer
  once orchestration is stable.
- Slash commands beyond `vibe-render` + `vibe-status`.
- Modes beyond the current three (dissertation/article/masters). Adding
  blog/talk/manual is a real opportunity but doesn't gate v0.1.
- A standalone CLI surface. Useful for CI, but the plugin is the
  product.
- Marketplace listing polish, telemetry, update channels.

**Naming + branding**: name decided 2026-04-26. ThesisStudio template's
persona pillars (Sourced Specificity, Disciplined Argument, Honest
Limits) are useful as a default for academic mode, but Vibe Thesis
should let users edit/replace pillars via bootstrap — they're not the
plugin's identity, they're a starting preset.

## 6. Open questions for Cartographer

These are the decisions that will shape the plugin and that this
sketch can't decide unilaterally:

1. **Plugin manifest format**: confirm the current Claude Code plugin
   spec (`plugin.json`? `claude-plugin.toml`? something else?). The
   sketch in §4 is shape-agnostic; the manifest is Cartographer's call
   based on the live docs at draft time.
2. **Mode taxonomy**: keep `THESIS_MODE` (dissertation/article/masters),
   or generalize to `VIBE_MODE` (thesis/blog/paper/talk/manual)? The
   numbered scaffold (00_…08_) was designed for academic work. How well
   does it fit a blog post or technical manual? If it doesn't, do we
   ship a *thesis-only* MVP and add other modes later, or rework the
   scaffold up front?
3. **How does Claude detect "I'm in a Vibe Thesis project"?** Options:
   presence of a `vibe-thesis.yaml` marker file, presence of the
   numbered scaffold, a stanza in `CLAUDE.md`, or all three. The
   marker affects how the orchestrator skill chooses scaffold-mode
   vs. iterate-mode.
4. **Sub-skill scope**: ship sub-skills *inside the plugin* (so they're
   available wherever Vibe Thesis is installed) vs. *templated into the
   user's repo* (so they're versioned with the user's content)? Current
   repo uses templated; plugin form might benefit from in-plugin so
   updates flow automatically. Trade-off: in-plugin means user can't
   easily customize per-project.
5. **Dev container — required or optional?** The toolchain works
   natively if the user has pandoc + texlive + node installed. Plugin
   can either require dev container (simpler support, narrower
   audience) or treat it as one of two install paths.
6. **Update path**: after a user scaffolds, how do plugin improvements
   reach them? A `/vibe-update` command that diffs templates against
   the user's tree and offers structured merges? Or assume one-shot
   scaffolding and rely on the user to merge by hand?
7. **Distribution + identity**: where does the plugin live? Personal
   GitHub (`estevanhernandez-stack-ed/vibe-thesis`)? An org
   (`@626labs/vibe-thesis`)? Distributed via Anthropic's plugin
   marketplace, raw git URL, or both?
8. **Cartographer's own scope**: produce the spec doc only, or also
   drive the first extraction commits? If the latter, **work in a new
   sibling repo `vibe-thesis/`, not in this article repo's tree** —
   this repo needs to keep functioning as an article repo for its
   primary use (Estevan finishing the agentic-architect article).

## 7. Where to start (suggested first three moves for Cartographer)

1. **Verify the Claude Code plugin format**. Read the official plugin
   docs and write a minimal "hello world" plugin that ships one skill
   + one slash command + one templated file. Confirm install/load
   path works on your machine before designing the real one.
2. **Audit `template/main` in this repo**. Run
   `git diff template/main..HEAD -- ':!03_BODY' ':!02_RESEARCH'
   ':!01_PLANNING' ':!05_CITATIONS' ':!08_OUTPUT'` to isolate
   infrastructure-only deltas vs. user content. Confirm the local
   contributions listed in §2 are the complete set.
3. **Create `vibe-thesis/` as a new sibling repo**, scoped to the
   plugin only. Copy the infrastructure files from §1 into a
   `templates/` payload, write a stub orchestrator `SKILL.md`, and
   validate the MVP boundary: install the plugin into a fresh
   directory, type "scaffold a vibe thesis project for me," and
   confirm the resulting tree round-trips through `npm run
   render:pdf`. If it works, the boundary is right. Iterate.
